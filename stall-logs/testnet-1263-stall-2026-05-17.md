## Immediate Stall Mechanism

The chain halted at height `23364` because height `23365` could not obtain the configured quorum of 4 validator votes.

Evidence:

- Validator 5 proposed height `23365` and logged:
  - `QC error - block proposal failed`
  - `Insufficient validator votes: 3 votes, 4 required for quorum`
- Validator 1 repeatedly refused vote requests for height `23365`:
  - `proposal height 23365 does not extend local tip 23363`
- Validator 4 repeatedly refused vote requests for height `23365`:
  - `proposal height 23365 does not extend local tip 23018`
- Validators 2, 3, and 5 were aligned at height `23364`, but that gives only 3 usable votes.

With Validator 4 far behind and Validator 1 missing height `23364`, the network had only three validators able to vote for `23365`. The consensus threshold remained 4, so every block proposal for `23365` failed.

## Catch-Up Failure

Validator 1 and Validator 4 both detected that peers were ahead and attempted catch-up, but the catch-up loop did not apply the missing blocks.

Validator 1 log pattern:

- local height: `23363`
- best validator height: `23364`
- error: `Timeout waiting for blocks up to height 23364`

Validator 4 log pattern:

- local height: `23018`
- best validator height: `23364`
- error: `Timeout waiting for blocks up to height 23082`

The relevant code path is:

- `src/consensus/consensus_algorithm.rs`
  - Before producing, validators call `get_best_validator_peer_height_with_support(...)`.
  - If the best validator height is ahead of local height, `sync_validator_to_network_tip(...)` calls `SYNC_MANAGER.start_sync()`.
- `src/sync/manager.rs`
  - Sync requests a reconciliation range starting at `local_height - 8`.
  - It waits for the target height to be applied.
  - If the target height is not reached before timeout, it returns `Timeout waiting for blocks up to height ...`.
- `src/p2p/networking.rs`
  - Incoming vote requests whose proposal does not extend the local tip trigger `request_vote_request_parent_sync(...)`.
  - The lagging validator waits only `900ms` for the parent to arrive before refusing the vote.

This means the runtime notices the lag, but the missing block batch is not applied quickly or reliably enough to make the validator eligible to vote before consensus moves on.

## Root-Cause Statement

The halt was caused by quorum loss after two validators became ineligible to vote for height `23365`:

1. Validator 4 fell behind to height `23018`.
2. Validator 1 failed to apply height `23364` and remained at height `23363`.
3. The remaining aligned validators, V2/V3/V5, could only provide 3 votes.
4. Chain 1263 requires 4 votes for quorum, so height `23365` could not finalize.
5. The built-in catch-up path repeatedly timed out and did not restore V1 or V4 to the network tip.

## Additional Runtime Cause Found

The on-disk validator data showed that V1 and V4 had appended duplicate same-height local fork blocks:

- V1 had canonical height `23363` hash `9548a05e6a16812cf5db8f4a2354377a73741b90f0ee7208014e37f2c92c61bb`, followed by another height `23363` hash `3033fa05ae135c1eaa747dc5f3765cc5ac7b5797f83150b1fffa93d2497d9419`.
- V4 had canonical height `23018` hash `4629582b771a1642f41b378b44ad2a6e142db932b6668c44505eba39611e9f74`, followed by another height `23018` hash `58c4a19753563c30c7d50d5e98c22802bd2d2571825895239374d209322b8917`.

This was not a genesis fork. It was a local same-height append race.

Source-level fix staged in the workspace:

- `src/block.rs` now has `BlockChain::add_block_extending_tip(...)`, which rejects a same-height fork or wrong-parent append and treats an exact duplicate tip as already applied.
- `src/consensus/consensus_algorithm.rs` now uses that guard after the vote wait, before applying a committed local proposal to the in-memory chain.

This addresses the race where consensus releases the chain lock while waiting for votes, P2P advances the local tip, and the proposer later tries to append a stale block that no longer extends the current tip.

## Topology Misconfiguration Found After Review

The intended TESTNET topology is:

- Genesis validators and observer live on the WireGuard private plane.
- Public nodes, including RPC gateway, explorer/indexer, bootnodes, and seed servers, are public-facing.
- Public nodes should communicate with validator/observer infrastructure through the two relayers.
- Relayers should expose a public-facing surface to public nodes and a VPN-facing surface toward validators/observer.

Live inspection found violations of that topology:

- Node-RPC and Node-EXP had been manually rewired to dial validators directly as persistent peers. That was an operator error during the rollout and bypassed the relayer tier.
- Validators accepted direct public connections from `74.208.227.23` (`genesisrpc` and `genesisindexer`) in their peer tables.
- Validator processes bind P2P, RPC, WS, and metrics to `0.0.0.0`; V1-V3 public IPs were reachable from outside on `5622`, `5640`, `5660`, and `6030`.
- The validator runtime TOML advertises `10.69.0.x` private P2P addresses, but the host firewall/listen surface still exposes validator services publicly.
- Relayer WireGuard handshakes observed from validators were stale by roughly two days, and validator probes to relayer private service ports such as `10.69.0.20:5622`, `10.69.0.20:5640`, `10.69.0.202:5622`, and `10.69.0.202:5640` were closed.

This means the validator setup was not enforcing the intended private-only validator surface, and the relayer tier was not available as the required bridge. The direct RPC/indexer peering was not the correct fix for relayer unavailability; it should be removed and replaced with a working relayer path.

This topology issue does not change the immediate consensus symptom: height `23365` failed because only three validators could vote. It does explain why public service nodes could attach directly to validators and why the deployment drifted away from the intended relay-mediated design.

## Remaining Follow-Up

Review and harden the catch-up behavior in `src/sync/manager.rs` and `src/p2p/networking.rs`, especially:

- why V1 did not receive/apply height `23364` from peers that had it;
- why V4 repeatedly timed out while requesting ranges from `23019` onward;
- whether vote-request parent sync should wait longer or synchronously request the exact missing parent from a peer known to have it;
- whether validators should refuse block production while any active genesis validator is known to be behind enough to break quorum.

Operationally, the next blocker is relayer access. Public RPC and Node-EXP are intentionally relayer-only now, but they cannot catch up until Relayer-1 and/or Relayer-2 are reachable and serving chain 1263 on their public and VPN surfaces.
