## Stall symptoms

- Validators 1, 2, and 4 looped at proposed height `55608` and failed quorum with `Insufficient validator votes: 3 votes, 4 required for quorum`.
- Validator 3 refused the V1/V2/V4 proposals because `proposal parent hash does not match local tip at height 55607`.
- Validator 5 refused height `55608` proposals because `proposal height 55608 does not extend local tip 55468`.
- Validator 5 also logged repeated catch-up failures: `Timeout waiting for blocks up to height 55532`.
- Validator 5 had repeated persistence backlog logs beginning earlier in the incident window: `Skipping chain persistence because a previous save is still running`.

## Root cause

The chain split because a local validator vote lock could be advanced to a different block at the same height when the new proposal used a higher round number.

The defective source path was `src/consensus/dual_quorum.rs`, in `DualQuorumConsensus::register_local_vote_intent`. The removed behavior logged `Advancing local vote lock to higher-round proposal`, then replaced the locked block hash, proposer, and latest round. That permitted one validator to sign or intend to sign a conflicting same-height block in a later round.

With a 5-validator set and a 4-vote quorum, overlapping same-height votes can certify two conflicting blocks:

- Block A can receive votes from V1/V2/V4/V5.
- Block B can then receive votes from V1/V2/V3/V4 if higher-round local locks are allowed to replace prior same-height locks.
- Both blocks have the same height and parent, but different validator/timestamp/transactions data, so they produce different hashes.

That pattern matches the live evidence. V5 logged that it had already locally voted for a different block at height `55468`, then stayed on its local height-55468 tip. Later, V3 was on a different height-55607 block from V1/V2/V4. At that point the network could not form the required 4-vote quorum for height `55608`.

## Why the hashes differed

The persisted `Block` hash is calculated from:

- `block_index`
- `previous_hash`
- `validator_id`
- `nonce`
- `timestamp`
- `transactions_root`

Therefore two blocks can share the same height and parent hash while still producing different block hashes if the proposer, timestamp, nonce, or transaction root differs. The height `55607` split shows exactly that: V1/V2/V4 and V3 shared the same parent but recorded different proposers/timestamps and therefore different block hashes.

## Source fix applied locally

`src/consensus/dual_quorum.rs` was updated so a validator's local vote intent remains locked to the first block hash it accepted for a given validator, epoch, and height. The same block hash may still be replayed or repeated in later rounds, but a different block hash at the same height is rejected even if the requested round is higher.

The regression test was updated to assert that higher-round conflicting same-height proposals are rejected:

- `local_vote_intent_rejects_same_height_conflict_across_rounds`

Validation run:

```text
cargo test --lib consensus::dual_quorum -- --test-threads=1
15 passed; 0 failed
```
