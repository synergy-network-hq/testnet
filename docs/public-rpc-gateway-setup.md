# Public RPC Gateway Setup — 74.208.227.23

**Purpose:** Stand up the public-facing Testnet-Beta RPC and WebSocket endpoints on a dedicated server, proxied through nginx with TLS.

After this is done, `https://testbeta-core-rpc.synergy-network.io` and `wss://testbeta-core-ws.synergy-network.io` will be live and pointing at this machine.

---

## Architecture

```
Internet
  │
  ├─ HTTPS :443 → nginx → 127.0.0.1:<assigned-rpc-port>  (RPC HTTP / JSON-RPC)
  └─ WSS   :443 → nginx → 127.0.0.1:<assigned-ws-port>   (WebSocket)

synergy-testbeta binary (rpc_gateway role)
  ├─ P2P     :assigned from the sequential `5622+` range
  ├─ RPC     :assigned from the `5640+` range (loopback only — nginx proxies it)
  └─ WS      :assigned from the `5660+` range (loopback only — nginx proxies it)
```

The node binary listens on loopback for RPC and WS. nginx handles TLS termination and proxies public traffic in. The assigned P2P listener port is the only node port that needs to be publicly reachable directly.

This gateway is not a validator. It should join P2P so it can discover peers and stay synced, but block production remains validator-only.

## Prerequisites

- SSH access to `74.208.227.23` (root or a user with sudo)
- Domain control over `synergy-network.io` to set DNS A records
- The `synergy-testbeta-linux-amd64` binary from the repo's `binaries/` folder
- The workspace folder generated by the control panel (see Step 4)

## Step 1 — Verify DNS Records

In your DNS provider (wherever `synergy-network.io` is managed), confirm these two A records already exist and still point at the RPC gateway host:

| Name | Type | Value | TTL |
|------|------|-------|-----|
| `testbeta-core-rpc` | A | `74.208.227.23` | 300 |
| `testbeta-core-ws` | A | `74.208.227.23` | 300 |

Verify they resolve before proceeding:

```bash
dig +short testbeta-core-rpc.synergy-network.io
dig +short testbeta-core-ws.synergy-network.io
# Both should return: 74.208.227.23
```

## Step 2 — Server Packages

SSH into the server and install dependencies:

```bash
ssh root@74.208.227.23

apt update && apt install -y nginx python3-certbot-nginx ufw
```

---

## Step 3 — Firewall

```bash
ufw allow 22/tcp        # SSH — do this first so you don't lock yourself out
ufw allow 80/tcp        # HTTP — needed for certbot ACME challenge
ufw allow 443/tcp       # HTTPS — the public RPC endpoint
ufw allow <assigned-p2p-port>/tcp  # P2P — node peer connectivity
ufw --force enable
ufw status
```

The assigned RPC and WS ports do **not** need to be open — nginx proxies them from 443.

## Step 4 — Provision the RPC Gateway Node (Control Panel)

Do this on your Mac, not on the server.

1. Go to the **Testnet-Beta Dashboard**
2. Click **Setup a New Node**
3. Select role: **RPC Gateway Node**
4. When the setup wizard asks about device details, click **Continue**
5. The wizard will detect your Mac's IP and then ask: *"This node type is built to run on a dedicated public server. Enter that server's IP address now."*
6. **Enter `74.208.227.23`** and press Enter (or click the button) — this gets baked directly into `node.toml` and `nginx.conf`
7. Confirm the workspace folder and click **Provision Node**

After setup, note the workspace path shown in the node detail view. It will look something like:

```
~/Library/Application Support/synergy-node-control-panel/testbeta/nodes/<node-id>/
```

The workspace will contain:
- `config/node.toml` — node configuration with the assigned listener, RPC, WS, discovery, and metrics ports baked in
- `config/peers.toml` — bootstrap peer list
- `config/aegis.toml` — security overlay
- `nginx.conf` — ready-to-deploy nginx reverse proxy config

---

## Step 5 — Verify node.toml Public Address

Open `config/node.toml` in the workspace and confirm the server's IP was picked up correctly:

```toml
[p2p]
public_address = "74.208.227.23:<assigned-p2p-port>"

[rpc]
bind_address = "0.0.0.0"
cors_enabled = true
cors_origins = ["*"]
```

The RPC bind address and CORS settings are set automatically for the `rpc_gateway` role. If `public_address` somehow shows your Mac's IP instead, edit it manually before transferring the workspace.

## Step 6 — Transfer Workspace and Binary to the Server

From your Mac, copy the workspace and the Linux binary:

```bash
# Set these to match your actual paths
WORKSPACE_PATH="$HOME/Library/Application Support/synergy-node-control-panel/testbeta/nodes/<node-id>"
SERVER="root@74.208.227.23"
BINARY="/absolute/path/to/synergy-testnet-beta/binaries/synergy-testbeta-linux-amd64"

# Create directory structure on server
ssh $SERVER "mkdir -p /opt/synergy/rpc-gateway /opt/synergy/bin"

# Copy workspace
rsync -av "$WORKSPACE_PATH/" $SERVER:/opt/synergy/rpc-gateway/

# Copy binary
scp "$BINARY" $SERVER:/opt/synergy/bin/synergy-testbeta
ssh $SERVER "chmod +x /opt/synergy/bin/synergy-testbeta"
```

---

## Step 7 — Deploy nginx Config

On the server:

```bash
# Create the ACME challenge directory certbot needs
mkdir -p /var/www/letsencrypt

# Deploy the generated nginx config
cp /opt/synergy/rpc-gateway/nginx.conf \
   /etc/nginx/sites-available/testbeta-core-rpc.synergy-network.io.conf

ln -sf /etc/nginx/sites-available/testbeta-core-rpc.synergy-network.io.conf \
       /etc/nginx/sites-enabled/

# Remove the default nginx site if it exists (avoids port 80 conflict)
rm -f /etc/nginx/sites-enabled/default

# Test and reload — this will work because the SSL blocks reference cert paths
# that don't exist yet; nginx will load in degraded mode for now.
# If nginx refuses to start due to missing cert, see the note below.
nginx -t && systemctl reload nginx
```

> **Note:** If `nginx -t` fails with "cannot load certificate" because the cert doesn't exist yet, temporarily comment out the two `ssl_certificate` and `ssl_certificate_key` lines in the 443 server blocks, reload nginx, then get the cert (Step 8), then uncomment and reload again.

---

## Step 8 — Get SSL Certificate

```bash
certbot --nginx \
  -d testbeta-core-rpc.synergy-network.io \
  -d testbeta-core-ws.synergy-network.io \
  --agree-tos \
  --non-interactive \
  --email support@synergy-network.io
```

Certbot will:
1. Complete the ACME HTTP-01 challenge using nginx
2. Issue the certificate and save it at `/etc/letsencrypt/live/testbeta-core-rpc.synergy-network.io/`
3. Automatically update the nginx config with the correct cert paths and reload nginx

After certbot finishes, verify:

```bash
nginx -t && systemctl status nginx
# Should show active (running)

curl -s https://testbeta-core-rpc.synergy-network.io/healthz
# Should return: ok
```

## Step 9 — Run the Node as a systemd Service

Create a systemd unit so the node starts on boot and auto-restarts:

```bash
cat > /etc/systemd/system/synergy-rpc-gateway.service << 'EOF'
[Unit]
Description=Synergy Testnet-Beta RPC Gateway Node
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=root
WorkingDirectory=/opt/synergy/rpc-gateway
ExecStart=/opt/synergy/bin/synergy-testbeta start --config /opt/synergy/rpc-gateway/config/node.toml
Restart=on-failure
RestartSec=10
StandardOutput=append:/opt/synergy/rpc-gateway/logs/node.log
StandardError=append:/opt/synergy/rpc-gateway/logs/node.log

[Install]
WantedBy=multi-user.target
EOF

mkdir -p /opt/synergy/rpc-gateway/logs

systemctl daemon-reload
systemctl enable synergy-rpc-gateway
systemctl start synergy-rpc-gateway
systemctl status synergy-rpc-gateway
```

## Troubleshooting

**nginx fails to start / "cannot load certificate"**
The SSL certificate doesn't exist yet. Follow the note in Step 7 to temporarily comment out the ssl_certificate lines, reload nginx, then run certbot.

**certbot ACME challenge fails**
- Confirm DNS A records resolve to 74.208.227.23: `dig +short testbeta-core-rpc.synergy-network.io`
- Confirm port 80 is open: `ufw status` and `ss -tlnp | grep :80`
- Confirm nginx is running and serving the acme-challenge location

**Node starts but shows no peers**
- Confirm the assigned P2P listener port is open in the firewall
- Check `config/peers.toml` has the correct bootnode and seed server entries
- Tail the node log for connection attempt details

**Node starts but RPC returns errors**
- Confirm the node has synced (check block height in logs)
- Check `config/node.toml` has `bind_address = "0.0.0.0"` under `[rpc]`
- Confirm nginx is proxying to the assigned RPC and WS ports from the generated workspace

**"public_address" in node.toml is wrong**
Edit `/opt/synergy/rpc-gateway/config/node.toml` directly on the server:
```toml
[p2p]
public_address = "74.208.227.23:<assigned-p2p-port>"
```
Then restart: `systemctl restart synergy-rpc-gateway`

## Port Reference

| Port | Protocol | Publicly open? | Purpose |
|------|----------|---------------|---------|
| 22 | TCP | Yes | SSH |
| 80 | TCP | Yes | HTTP (certbot ACME + redirect to HTTPS) |
| 443 | TCP | Yes | HTTPS — RPC and WS via nginx |
| `5622+` | TCP | Yes | Sequential node listener / P2P assignment |
| `5640+` | TCP | **No** | Local RPC upstream assignment |
| `5660+` | TCP | **No** | Local WebSocket upstream assignment |
| `5680+` | TCP | Optional | Local discovery assignment |
