Executive Summary
This document is the authoritative technical specification for every theft-mitigation measure that Synergy Network will implement at the protocol, language, and wallet layers. It is organized around the thirteen major attack categories documented in the Synergy Network Cryptocurrency Theft Analysis, and expands each countermeasure into a concrete implementation plan with step-by-step engineering tasks and verification methodology that the development team can execute directly.
The central thesis driving every decision in this document is that most cryptocurrency theft does not exploit bugs in individual contracts — it exploits architectural assumptions inherited from Ethereum's original design. Unlimited approvals exist because ERC-20 was designed before DeFi existed. Blind signing persists because transaction formats predate complex dApp interactions. Maximal extractable value (MEV) is structural to public mempools. Reentrancy is possible because the EVM allows arbitrary external calls mid-execution.
Synergy Network addresses this at the root. Rather than layering countermeasures on top of an insecure base, the protocol is designed so that the insecure patterns are structurally impossible. This document explains precisely what that means in engineering terms: what is being built, at which layer, how it eliminates each threat class, and how the team will verify that the implementation actually works under adversarial conditions.

Architecture Context: Three Defence Layers
Every countermeasure in this document operates at one or more of three distinct defence layers within the Synergy stack. Understanding which layer is responsible for which defence is essential because failures in one layer must not cascade to others — this principle of fail-closed, plane-isolated containment is the architectural thread that runs through the entire system.
Layer 1 — L1 Protocol (Consensus Layer)
The consensus layer enforces rules that cannot be overridden by any application, wallet, or smart contract. Defences implemented at this layer are universally and automatically applied: time-delayed transfers, encrypted mempools, native naming systems, approval expiry enforcement, rate limiting, and cross-chain proof verification all live here. If an action violates a consensus-layer rule, it is rejected unconditionally before execution.
Layer 2 — SynQ Language (Compiler and VM)
SynQ is a purpose-built smart-contract language whose type system and compiler eliminate entire vulnerability classes by making them inexpressible. Linear asset types make reentrancy structurally impossible. Bounded loops prevent gas-exhaustion attacks. The absence of raw delegatecall eliminates storage-collision exploits. These are compile-time guarantees, not runtime checks. A contract that would be vulnerable cannot compile.
Layer 3 — Wallet Architecture (Three-Plane Model)
The Synergy Wallet is a protocol component, not a peripheral application. It enforces the Local Policy Engine as a hard pre-signature gate, separates Identity, Routing, and Execution into isolated planes, and requires guardian-mediated recovery with time-locks. Compromise of the wallet does not imply compromise of identity or authority because the architecture explicitly decouples the two.
KEY PRINCIPLE:  Defences are additive across layers. An attack that bypasses a wallet-layer control still faces protocol-layer enforcement. An attack that bypasses the protocol layer is blocked by the SynQ type system. No single layer is the last line of defence.
Attack Category 1: ERC-20 Approval and Allowance Exploits
1.1 Threat Description
The ERC-20 approve() / transferFrom() model was designed for a world where one party occasionally delegates spending authority to another. In practice, dApps request unlimited (2^256 − 1) approvals for convenience and users grant them without understanding the long-lived attack surface this creates. An attacker who compromises a contract — through a frontend injection, a supply-chain attack, or a protocol upgrade — can then call transferFrom() to drain every wallet that has an outstanding approval, even if those wallets have never been touched again.
The BadgerDAO incident ($120M, December 2021) is the canonical case: attackers compromised a Cloudflare API key, silently harvested increaseAllowance() approvals over 12 days, and then drained all 500 affected wallets in a single operation. Crucially, no private key was ever compromised. The approval mechanism itself was the vulnerability.
1.2 Synergy Implementation: Mandatory Scoped Approvals
SynQ eliminates unlimited approvals entirely. Every approval is a protocol-enforced struct — it cannot be expressed in any other form — and the struct requires five mandatory fields. The protocol consensus layer rejects any approval message that omits any of these fields.
Field
spender (address), amount_cap (u256), expiry_timestamp (u64), function_selector (bytes4), parameter_constraints (policy object)
No Expiry
Rejected at consensus — transactions containing approval structs without expiry_timestamp are invalid and will never be included in a block
revokeAll() type
A native transaction type (not a contract function) that atomically clears every outstanding approval from a given account in a single consensus operation
Max Approval Cap
Protocol-configurable upper bound on amount_cap per approval type; governance can set per-token or per-protocol global caps
Approval Audit Log
All approval creations, uses, and expirations are emitted as indexed consensus-level events, permanently queryable without relying on any contract's own event emission

Attack Category 2: Permit and Permit2 Signature Phishing
2.1 Threat Description
EIP-2612 permit() and Uniswap's Permit2 enable gasless, off-chain signature-based approvals. The attack flow is devastatingly simple: a drainer website queries the victim's balances, generates permit messages, presents them as benign 'sign this message' requests, captures the signatures, and immediately submits them on-chain. The victim pays no gas, sees no on-chain trace, and most wallets cannot simulate the outcome because the operation is off-chain until the attacker chooses to submit it.
In H1 2024, 260,000 victims lost $314M to permit-based phishing — a figure already exceeding the entire previous year. Permit and Permit2 signatures accounted for 38% of cases exceeding $1M. The scale demonstrates that off-chain signature models are incompatible with user safety at scale.
2.2 Synergy Implementation: Protocol-Native Intent System
Synergy eliminates the off-chain signature attack vector at the protocol level by replacing it with a mandatory on-chain intent submission model. No approval-equivalent operation can be authorized through an off-chain message alone.
Core Change
All approval-equivalent operations MUST be submitted as on-chain transactions. Off-chain messages are scoped by the protocol to read-only operations only and cannot authorize any token transfer or state mutation.
simulate() opcode
A native simulate() opcode is available in the SynQ VM. Every transaction must pass simulation before it can be submitted. The simulation result is a human-readable asset flow description (e.g., 'you will transfer 100 USDC to address X and receive 0.05 ETH') that must be presented to the user before signing.
Off-chain signature scope
Off-chain EIP-712-style signatures are restricted at the protocol level to operations flagged as READ_ONLY in the SynQ function declaration. Any attempt to use an off-chain signature to authorize a state-mutating operation is rejected by the consensus layer.
No gasless drain path
Because permits require on-chain submission, the attacker must pay gas. This does not eliminate the threat but eliminates the 'zero trace until too late' property that makes permit phishing uniquely dangerous.

Attack Category 3: Transaction Signing Tricks and Ice Phishing
3.1 Threat Description
Ice phishing encompasses all attacks that manipulate users into signing malicious transactions — not by stealing their keys, but by deceiving them about what they are signing. This includes eth_sign (signing arbitrary hashes with no human-readable content), SetApprovalForAll (granting complete NFT portfolio control in one signature), and most dangerously, transaction simulation spoofing — where a contract behaves safely during simulation but maliciously during actual execution by checking block.number or block.timestamp. A user lost 143.45 ETH (~$461K) to simulation spoofing in January 2025. The 'bitflip attack' used by Vanish Drainer on Solana shows victims receiving tokens in simulation while their wallets are drained in execution.
3.2 Synergy Implementation: Mandatory Human-Readable Manifests + Simulation Spoof Detection
Synergy addresses ice phishing through two reinforcing mechanisms: protocol-enforced mandatory transaction manifests and simulation-execution divergence detection.
Mandatory Manifests
Every contract call must include NatSpec-equivalent descriptions compiled into transaction metadata by the SynQ compiler. These manifests describe what the function does in plain language. The protocol compares manifest descriptions against simulation results and flags mismatches before presenting the signing UI.
No Raw Hash Signing
eth_sign-equivalent raw hash signing is not supported at the protocol level. All signing operations produce structured, type-safe authorization objects with explicit scope declarations. This eliminates the entire class of 'sign this opaque hash' phishing.
Spoof Detection
The simulation engine runs at two different block heights (current and current+1) and checks for execution divergence. Any contract that produces different results based on block.number or block.timestamp within the signing window triggers a SIMULATION_DIVERGENCE warning that must be acknowledged before signing.
SetApprovalForAll equiv.
SynQ's capability system requires NFT-collection approvals to enumerate the specific token IDs being delegated or to use a scoped delegation with an explicit cap. Blanket 'approve all' semantics are not expressible in SynQ.

Attack Category 4: Signature Replay Attacks
4.1 Threat Description
Replay attacks reuse signatures valid in one context in a different context. Cross-chain replay (exploited in the $20M Optimism OP token theft, June 2022) occurs when chain IDs are absent from signature hashes. Same-chain replay occurs when contract authorization omits nonces. ECDSA nonce reuse — where the same random k is used in two signatures — allows complete private key extraction. A December 2024 arXiv paper demonstrated that EIP-7702 delegation signatures can be replayed across every EVM chain using the same EOA, since each chain treats the authorization as fresh.
4.2 Synergy Implementation: Mandatory Scoped Signature Fields + Deterministic Nonce Generation
Synergy requires every signature in the system to carry five mandatory context fields that make replay across chains, contracts, functions, or time intervals structurally impossible.
Mandatory fields
{chain_id: u64, contract_address: bytes32, function_selector: bytes4, nonce: u64, expiry: u64} — all five fields are required and enforced at the consensus layer for every authorization signature
Chain-ID binding
The chain_id in every signature is validated against the consensus state at admission time. A signature from Synergy testnet cannot be replayed on mainnet because the chain_id values differ and both are checked.
Nonce management
Nonces are managed at the consensus layer, not by individual contracts. The per-account nonce is a global monotonic counter maintained in the Synergy state trie. No contract can reset, replay, or manipulate nonces.
Deterministic nonce gen
The Aegis cryptographic library enforces RFC 6979 deterministic nonce generation for all ECDSA-family signatures. This makes k-reuse mathematically impossible regardless of RNG quality, eliminating the entire class of private-key extraction via nonce reuse.
Expiry enforcement
Signatures with expiry timestamps in the past are rejected at the protocol level. Default maximum signature validity window is configurable via governance but starts at 24 hours for standard operations and 1 hour for high-value operations.

Attack Category 5: Sweeper Bots
5.1 Threat Description
Once a private key is compromised, sweeper bots monitor the mempool for any incoming transfer to the compromised address and instantly broadcast a competing transaction with higher gas to drain the funds before the owner can react. Advanced sweepers operate across multiple chains, sweep ERC-20 tokens based on live price feeds, and run indefinitely across all accounts derived from a compromised seed phrase. Recovery using Flashbots bundles is technically possible but requires expert knowledge, precise gas calculation, and even then is uncertain. Most users simply lose everything.
5.2 Synergy Implementation: Tiered Time-Delayed Transfers + Watchdog Accounts + Panic Mode
Synergy defeats sweeper bots structurally: if funds cannot be instantly transferred, sweeper bots cannot instantly drain them. Time delays are enforced at the consensus layer and cannot be bypassed by any application or smart contract.
Tier 1 (< $100 equiv.)
Instant execution — no delay. Standard microtransactions are not impeded.
Tier 2 ($100 – $10,000)
1-hour cancellation window. The transfer is committed to the ledger in a PENDING state but the assets are not released to the recipient for 3,600 seconds. During this window, the sender (or an authorized watchdog) can broadcast a CANCEL_TRANSFER transaction.
Tier 3 (> $10,000)
24-hour delay with guardian notification. An on-chain notification is automatically sent to all registered guardian accounts when a Tier 3 transfer is initiated. Any registered guardian or the watchdog account can veto.
Watchdog Account
A veto-only account that holds zero funds but has the authority to broadcast CANCEL_TRANSFER or FREEZE_ACCOUNT for its associated primary account. The watchdog key should be stored offline or on a separate hardware device. The watchdog cannot itself initiate transfers — only cancel or freeze.
Panic Mode
A PANIC_FREEZE transaction type that triggers via a separate offline panic key. When broadcast, it instantly freezes all outgoing transfers from the account for a configurable duration (default: 72 hours). The panic key is not stored in the wallet and is designed for use only when active compromise is detected.
Account Freeze RPC
rpc_freezeAccount and rpc_cancelPendingTransfer are first-class RPC methods that must be serviced by all nodes within 500ms, with higher priority than standard transactions.

Attack Category 6: Address Poisoning
6.1 Threat Description
Attackers send tiny transactions from addresses that match the first and last characters of the victim's regular recipients, exploiting the habit of copying addresses from transaction history. Academic research at USENIX Security 2025 documented 270 million attack attempts across Ethereum and BSC targeting 17 million unique addresses, resulting in $83.8M in confirmed losses. A single attacker group spent over $8M on GPU-powered vanity address generation. The $68M WBTC theft (May 2024) was the single largest incident. Testing found that 53% of Ethereum wallets fail to detect poisoning attempts.
6.2 Synergy Implementation: Protocol-Native Naming System + Anti-Spoofing Rules
Synergy eliminates address poisoning by making raw hex addresses secondary to human-readable identifiers at the protocol level. Users interact with names, not addresses.
Universal Account Names
Every Synergy account has a mandatory protocol-native human-readable identifier (SynID), similar to ENS but enforced by the consensus layer rather than a contract. SynID registration is required for accounts that initiate transactions above a minimum threshold. The wallet UI never displays raw hex as the primary identifier.
Address Book at Consensus
The protocol maintains per-account address books in the state trie. When a transaction is sent to a known contact, the wallet displays their SynID. When sent to an unknown address, the wallet displays a prominent UNKNOWN RECIPIENT warning.
Anti-Spoofing Rules
The consensus layer maintains a character-similarity distance metric (Levenshtein distance) for all registered SynIDs. Any attempt to register a SynID with a distance below a threshold from an existing SynID is rejected. New accounts attempting to send transactions to addresses they have never previously transacted with and whose hex representation is similar to known contacts trigger a POTENTIAL_POISONING warning.
VDF Address Cost
A Verifiable Delay Function (VDF) is required for vanity address mining — any address generation that involves trying more than a governance-configurable number of candidates requires completing a VDF proof. This makes GPU-scale vanity address grinding economically prohibitive.
QR-Code Default
The reference wallet defaults to QR-code address transfer rather than copy/paste. A QR code is generated from the SynID (not the raw hex) and the recipient's wallet verifies the SynID matches the expected account before completing the transfer.

Attack Category 7: Clipboard Hijacking Malware
7.1 Threat Description
Clipboard hijackers are malware families that silently replace copied cryptocurrency addresses with an attacker's address. Laplas Clipper (2022) was offered as a subscription service at $549/year and supported 19+ cryptocurrencies. ClipBanker (2025) was distributed via fake Microsoft Office add-ins on SourceForge and monitored 2.3 million addresses. Because clipboard monitoring is invisible to the user and operates at the OS level, it bypasses all in-wallet security checks. Binance reported a significant spike in clipper attacks in August 2024.
7.2 Synergy Implementation: Protocol Naming + Clipboard Integrity Verification
Because the SynID naming system makes raw hex addresses secondary, clipboard hijacking becomes significantly less effective — you cannot easily substitute a SynID without the victim noticing. Additionally, the reference wallet implements clipboard integrity verification as a default behavior.
SynID as primary identifier
Since users copy-paste SynIDs (human-readable) rather than hex addresses by default, clipboard substitution of a hex address has no effect when the recipient verification flow checks against the intended SynID.
Address integrity check
When a user pastes any identifier into the wallet's recipient field, the wallet re-fetches the expected address from the intent source (address book entry, dApp-provided link, QR code) and performs a hash comparison against the pasted content. Any mismatch triggers a CLIPBOARD_MISMATCH warning.
QR-Code-first UX
The reference wallet's default UX for address transfer is QR code scanning, which completely bypasses the clipboard. The wallet generates a QR code containing the SynID (not the hex address). The receiving wallet scans and resolves the SynID. No clipboard involvement.
Intent Source Anchoring
When a dApp or external link initiates a transaction, the wallet records the intended recipient from that initiating source and compares it against what is present in the transaction object at signing time. If they differ, the wallet refuses to sign.

Attack Category 8: Smart Contract Exploits
8.1 Threat Description
This is the broadest and most technically sophisticated category, covering reentrancy attacks (DAO hack, $60M; Curve Finance, $62-73M), flash loan governance attacks (Beanstalk, $182M; Euler Finance, $197M), MEV and sandwich attacks ($370-500M extracted from Solana users in 16 months), bridge exploits (Ronin, $624M; Wormhole, $325M), and delegatecall storage collisions (Parity Wallet #2, $154M permanently frozen). Flash loan attacks constituted 83.3% of eligible exploits in 2024.
8.2 Synergy Implementation: Multiple Structural Defences in SynQ and at the Consensus Layer
Reentrancy — Linear Asset Types
SynQ uses a linear asset type system inspired by the Move language. Assets are values that cannot be copied or dropped — they can only be moved from one owner to another. The SynQ compiler enforces that all asset references in a function must be fully consumed or stored before any external call is made. If a contract makes an external call while holding a live asset reference, compilation fails. This is not a mutex check at runtime — it is a type error at compile time. Reentrancy is inexpressible in SynQ.
Flash Loan Governance Attacks — Hot Potato Pattern + Holding Periods
SynQ enforces the 'hot potato' pattern for flash loan objects. Flash loan structs are defined without a 'drop' ability — they must be consumed by a repayment function before the transaction ends. More fundamentally, the governance module enforces a minimum token holding period for governance participation. Tokens acquired in the same block as a governance vote cast cannot be counted toward quorum. This makes single-block governance takeover structurally impossible, regardless of flash loan availability.
MEV and Sandwich Attacks — Encrypted Mempool + Fair Ordering
The Synergy protocol implements an encrypted mempool using threshold encryption. Transactions are encrypted with a shared public key before entering the mempool. They are decrypted only when a sufficient threshold of validators commit to the current transaction ordering. This means no validator or searcher can see transaction content before the ordering is finalized, eliminating the information asymmetry that makes frontrunning and sandwich attacks possible. A native MEV redistribution mechanism returns any detected extracted value to the transaction originators via a protocol fee adjustment.
Bridge Security — Native Cross-Chain Verification
SXCP provides native bridgeless cross-chain communication with protocol-level verification. Unlike bridge protocols that hold custodied funds in a pooled vault (the target of the Ronin and Wormhole hacks), SXCP creates no custody domain. Threshold signatures require geographically distributed validators. Mandatory time delays for large cross-chain transfers prevent instant drain scenarios even if validators are compromised.
Delegatecall — Explicit Type-Checked Interfaces
SynQ does not expose raw delegatecall. Module composition uses explicit, type-checked interfaces. Storage layouts are enforced at the compiler level — any module upgrade that changes the storage layout of an existing slot is a compilation error. Storage collision exploits are inexpressible in SynQ.

Attack Category 9: Social Engineering and Drainer-as-a-Service
9.1 Threat Description
The industrialization of crypto phishing through Drainer-as-a-Service platforms has created a thriving criminal economy. Inferno Drainer held 40-45% market share in 2024, taking a 30% affiliate cut. The DaaS model operates like legitimate SaaS: developers maintain the malicious infrastructure, affiliates deploy phishing sites and drive traffic. North Korea's Lazarus Group, responsible for $1.34B in crypto theft in 2024 including the Bybit hack ($1.5B, February 2025), uses fake LinkedIn job offers as their primary vector. AI-powered deepfake scams caused $4.6B in losses in 2024. The Ledger Connect Kit supply chain attack (December 2023) compromised 30,000+ browser downloads.
9.2 Synergy Implementation: Protocol Defences Collectively Dismantle the DaaS Model
The DaaS business model depends entirely on the signing step being opaque and irreversible. Synergy's combination of mandatory simulation, human-readable manifests, time-delayed transfers, and scoped approvals fundamentally undermines the model: if users can clearly see what they are signing and can cancel within a window, the drainer's attack surface collapses.
Supply chain protection
SynQ compiler requires formal verification for all contracts above a configurable value threshold at deployment time. Contracts that fail formal verification cannot be deployed to mainnet, regardless of how they reached the network. This prevents backdoored smart contract libraries from being deployed.
Frontend independence
Because Synergy's security model does not depend on the correctness of frontend UIs (the protocol layer enforces simulation, manifests, and time delays regardless of what the UI shows), DNS hijacking and CDN compromise attacks cannot cause fund loss — the protocol layer will still require genuine user confirmation of the correct manifest.
Phishing site ineffectiveness
Without the permit/off-chain signature vector (blocked by Category 2 implementation) and with mandatory on-chain simulation, phishing sites cannot drain wallets through a single signature. Every drain attempt requires an on-chain transaction that the time-delay system can block.
Verification requirement for high-value deployments
Contracts deploying above a value threshold must provide a SynQ formal verification proof as part of the deployment transaction. The consensus layer runs the proof verifier and rejects unverified deployments. This is implemented in the SynQ compiler's --verify-deploy flag.

Attack Category 10: Multicall and Batch Transaction Abuse
10.1 Threat Description
The ERC-2771 + Multicall vulnerability (December 2023) allowed attackers to spoof any address by crafting calldata where the last 20 bytes of each sub-call contain the victim's address. The TIME token exploit used this to burn 62 billion tokens from a Uniswap pool. A separate class exploits msg.value reuse in delegatecall-based multicall: an attacker sends 1 ETH but executes 10 swaps each reading msg.value = 1 ETH — effectively spending 10 ETH worth of value with 1 ETH of input.
10.2 Synergy Implementation: Linear Payment Types + Per-Sub-call Manifests
Linear payment values
In SynQ, msg.value (or its equivalent) is a linear type. It is consumed on first use within a transaction. Any attempt to read msg.value a second time within the same transaction returns zero — the type system prevents reuse. msg.value reuse is a type error.
msg.sender integrity
SynQ's authorization model eliminates the concept of calldata-based msg.sender spoofing. msg.sender is resolved from the transaction's cryptographic authorization chain at the consensus layer, not from calldata. No calldata manipulation can substitute a different sender identity.
Per-sub-call manifest display
Batch transactions display each sub-call's human-readable manifest separately in the wallet UI. The user sees an itemized list of every operation in the batch. Batches containing any sub-call with a missing or anomalous manifest are flagged before signing.
Batch size limit
Protocol-level maximum for batch transaction sub-call count (governance-configurable, default 20). Batches exceeding this limit are rejected at mempool admission.

Attack Category 11: Upgradeable Proxy Exploits
11.1 Threat Description
Proxy patterns separate state from logic to enable upgrades, but admin key compromise enables malicious upgrades that drain every user who has ever granted approvals. Access control failures in upgradeable contracts accounted for $953.2M in losses in 2024 — the single largest vulnerability category. The Wormhole exploit ($325M) used a deprecated, unprotected function in an upgradeable contract. 11.2 Synergy Implementation: Protocol-Enforced Upgrade Governance + Automatic Approval Invalidation
Time-locked upgrade proposals
Contract upgrades require a time-delayed governance proposal. The default delay is 7 days. The proposal must include: the new contract bytecode hash, a diff of the storage layout changes, and a formal verification proof for the new implementation.
Multi-party upgrade approval
Upgrades require multi-party authorization using the governance quorum mechanism. No single admin key can authorize an upgrade. The threshold and quorum are set by governance and enforced by the consensus layer.
Automatic approval invalidation
When a contract upgrade is finalized, the protocol consensus layer automatically invalidates all outstanding user approvals for the upgraded contract address. Users must explicitly re-approve the new version. This is non-negotiable and cannot be disabled by the contract deployer.
Compiler-enforced storage layout compatibility
The SynQ compiler's upgrade mode verifies storage layout compatibility between the current and proposed new implementation. Any slot that changes type, shifts position, or is removed is flagged as an upgrade error. Slot additions at the end of the layout are the only permitted upgrade change.

Attack Category 12: Key Extraction and Private Key Compromise
12.1 Threat Description
Private key compromise accounts for 80.5% of the largest crypto losses in 2024. Vectors include fake wallet apps (including a Chrome extension that exfiltrated seed phrases via fake Sui blockchain transactions), mass phishing emails, hardware wallet supply chain attacks (pre-seeded or firmware-tampered devices), and the Lazarus Group's fake LinkedIn job offer technique that produced the Bybit hack ($1.5B). The defining characteristic of this attack class is that the damage is instantaneous and typically irreversible: an attacker with a private key can drain everything within seconds.
12.2 Synergy Implementation: Social Recovery + Key Rotation + Hierarchical Keys + Factor-Bound Root Authority
The Synergy Wallet Architecture section of the project documentation establishes a foundational principle: 'seed phrase compromise alone does not constitute a security failure.' This is enforced by the Factor-Bound Root Authority property — the seed phrase is an entropy component, not a master key. Operational spending authority requires a second independent high-entropy factor not derivable from the seed phrase.
Factor-Bound Root Authority
The mnemonic-derived master seed combined with a second independent factor (hardware-backed or separately stored) is required for spending authority. An attacker who obtains only the seed phrase cannot spend. An attacker who obtains only the second factor cannot spend. Both are required.
Social Recovery (M-of-N)
The protocol-native recovery system uses an M-of-N guardian quorum. Guardians are other UMA identities. A 72-hour guardian modification delay prevents an attacker who briefly controls the account from replacing guardians before recovery can be initiated.
Key Rotation Without Address Change
Inspired by Aptos's authentication key model, Synergy allows the authentication key bound to a UMA identity to be rotated without changing the account's public address or SynID. An attacker who extracts a key can be shut out by key rotation without requiring fund migration.
Hierarchical Permission Levels
Three key tiers: (1) Daily spending key — low value operations only, policy-enforced caps; (2) High-value key — operations above policy thresholds, requires additional confirmation; (3) Recovery key — stored offline, used only to initiate recovery workflows. Compromise of the daily spending key does not grant high-value or recovery authority.
PQC Migration
The key rotation mechanism is designed to support migration from classical (ECDSA-equivalent) to post-quantum (ML-DSA / Aegis suite) without identity disruption. Keys can be rotated to PQC algorithms before quantum threats become practical.

13.1 Threat Description
Several emerging attack classes exploit newer protocol features or governance mechanisms. CREATE2 equivalent

The reference wallet verifies all RPC state proofs before using query results.
ERC-4337 equivalent
Synergy's account abstraction equivalent uses SCETP ingress with explicit authority chains rather than bundler-mediated UserOperations. Every execution request carries a cryptographic provenance chain traceable to a UMA identity. Bundler-type relayers cannot forge authority.

Attack Category 14: EIP-7702 Delegation Exploitation
14.1 Threat Description and Why This Is Categorically Different
EIP-7702 was activated on Ethereum mainnet as part of the Pectra upgrade in May 2025. It introduces a new transaction type (type 0x04) that allows an Externally Owned Account (EOA) to temporarily set its bytecode to point at a smart contract's code for the duration of a transaction — effectively letting any wallet become a smart account without redeployment. The legitimate use case is account abstraction: giving standard wallets programmable features like transaction batching, gas sponsorship, and social recovery.
The attack surface this creates is categorically more dangerous than every other phishing vector in this document combined. Every other category in this specification exploits a permission to do one specific thing — drain an approved token, replay a specific authorization, execute a known function. EIP-7702 delegation exploitation gives an attacker the ability to replace the victim's account with attacker-controlled contract code. That is not 'steal my tokens.' That is 'become me, permanently, across every chain I exist on, at zero ongoing cost.'
The five distinct attack vectors within this category are described below. Each builds on a real, demonstrated capability and together they represent the most significant new attack surface introduced to the EVM ecosystem since Permit2.
Vector 1 — Delegation Signature Phishing (Account Hijack in One Signature)
An EIP-7702 authorization tuple has the form (chain_id, delegate_address, nonce). To an unsuspecting user, a request to sign this tuple looks identical to the permit-style 'sign this message' requests they have been conditioned to accept for gasless transactions. When the attacker broadcasts the signed tuple as part of a type-0x04 transaction, the victim's EOA is delegated to execute the attacker's contract code for all subsequent operations. Unlike a permit signature that authorizes a single transfer, a successful delegation turns the victim's EOA into an attacker-controlled execution environment. Every future transaction the EOA participates in — including receiving tokens, interacting with protocols, or executing governance votes — now runs through the attacker's injected code.
Vector 2 — Cross-Chain Metastasis via chainId=0
A December 2024 arXiv paper demonstrated that EIP-7702 authorization tuples signed with chain_id=0 are treated as valid by every EVM chain where the signing address exists. This means a single phishing signature simultaneously compromises the victim's account on Ethereum mainnet, all EVM-compatible L2s (Arbitrum, Optimism, Base, Polygon), any sidechains, all testnets, and any future EVM chain the address is later used on. The victim does not need to interact with those chains — the delegation propagates automatically. The practical consequence: one successful phishing interaction can produce losses across every chain simultaneously, with the attacker invoking the delegated code wherever assets exist.
Vector 3 — Persistent Execution Takeover Beyond the Transaction
In some EIP-7702 implementation interpretations, the code delegation persists beyond the initiating transaction depending on whether the delegated contract resets or preserves the code slot. This creates an ongoing execution takeover where the attacker's code runs every time the compromised EOA is the execution context. This includes being called by other smart contracts that have legitimate approved interactions with the EOA — protocols that approved the EOA for a specific operation may find themselves invoking attacker-controlled logic instead.
Vector 4 — Paymaster-Funded Unlimited Exploitation
When combined with ERC-4337's account abstraction model, EIP-7702 creates a particularly dangerous amplification: paymasters can sponsor gas costs for bundled UserOperations targeting the compromised account. Because the attacker pays no gas, they can submit an unlimited number of exploitation attempts against the delegated account at zero cost. This turns account compromise from a one-time event into an ongoing, economically unbounded attack loop. Existing rate-limiting approaches based on gas cost economics become ineffective because the cost signal is completely severed.
Vector 5 — Social Engineering via 'Account Upgrade' Framing
EIP-7702's legitimate use case — enabling 'smart account' features for standard wallets — creates a powerful social engineering framing that attackers have already exploited. Phishing sites present delegation requests as 'upgrade your wallet to access advanced features,' 'enable gasless transactions,' or 'connect to our exclusive DeFi protocol.' Because the underlying technology is real and desirable, users who understand EIP-7702 are actually more susceptible to this framing, not less. The ability to distinguish a legitimate account upgrade delegation from a malicious one requires reading the delegate_address and verifying the delegated contract's code — a task that is completely beyond typical users and most security tooling.
14.2 Why Synergy Is Structurally Immune
Synergy's architecture eliminates EIP-7702-style delegation attacks through a combination of architectural decisions that predate EIP-7702's existence but happen to be exactly the right defenses against it. This is not coincidence — it reflects that EIP-7702's vulnerabilities arise from the same root cause that motivated Synergy's design: the collapse of identity, routing, and execution into a single key-controlled surface.
No EOA Code Slot Exists
Synergy's account model does not have an EIP-7702 equivalent mechanism. There is no 'code slot' attached to an identity that can be overwritten by a signed tuple. The three-plane wallet model separates Identity, Routing, and Execution by design. The Identity Plane never executes transactions and has no code slot to delegate. Delegation-style attacks have no surface to target at the account model level.
chainId=0 Explicitly Rejected
Synergy's AuthorizationSignature type (Category 4 implementation) requires chain_id to be a non-zero value matching the current chain. The consensus layer rejects any authorization where chain_id is zero. The cross-chain metastasis attack vector (Vector 2) does not exist in Synergy because its prerequisite — a chain_id=0 wildcard — is a protocol-level invalid state.
UMA Identity Decoupling
In Synergy, identity is not equivalent to key material. Even if an attacker could obtain a delegation-equivalent signed authorization, it would not grant them the victim's UMA identity, which is the actual authorization substrate. The UMA identity governs recovery rights, governance weight, contract authority, and policy state. Delegating a key does not delegate the identity, so the attacker gains execution capability for specific operations but not full account control.
Local Policy Engine as Hard Gate
Any operation that modifies execution authority — which a delegation request inherently does — must pass through the Local Policy Engine before any signing occurs. The Policy Engine's default rules classify execution authority modification as a high-value operation requiring the high-value key tier, explicit guardian notification, and a mandatory 24-hour delay. An attacker cannot trick a user into signing a delegation through a phishing site because the policy gate blocks it before the signing UI is ever reached.
Mandatory Delegation Manifest
The SynQ compiler and wallet policy engine treat 'execution delegation' as a class of operation that requires a human-readable manifest explicitly stating: the delegate contract address, the scope of delegated authority, the duration, and the exact operations the delegate can perform on the user's behalf. The manifest must be confirmed by the user reading it — not just clicking Accept.
SCETP Replaces Bundler Model
Synergy's account abstraction uses SCETP ingress with explicit cryptographic authority chains rather than ERC-4337 bundler-mediated UserOperations. There is no paymaster concept in Synergy's native execution model that can sponsor unlimited free attack attempts. Every execution request traces to a UMA identity via a verifiable chain. The paymaster-funded unlimited exploitation vector (Vector 4) has no entry point.

Overall Verification Framework
Security implementation without systematic verification is theater. The following framework defines how the engineering team will verify that every security measure in this document actually works under adversarial conditions — not just under ideal conditions. The framework has four levels.
Level 1 — Unit Tests (Per-Feature)
Every feature described in this document must have associated unit tests that confirm the basic enforcement behavior. These are specified within each attack category section above. Unit tests must be automated and run as CI gate checks on every commit to the protocol repository. No code may merge to main that reduces the unit test pass rate below 100% for security-critical test suites.
Level 2 — Adversarial Integration Tests (Per Attack Category)
Each of the 13 attack categories must have an adversarial test suite that attempts the actual attack flow — not just the mitigation. For example, the reentrancy test suite must deploy a contract that genuinely attempts reentrancy, confirm it cannot compile in SynQ, and (for the EVM compatibility lane) confirm that the protocol-level defenses prevent execution even if the attack is expressed via the EVM compatibility interface. The goal is to confirm that every attack described in the taxonomy fails completely.
Level 3 — Formal Verification (Protocol Invariants)
The following properties must be formally proven before mainnet launch: (1) linear asset conservation — no SynQ execution path can create or destroy assets without explicit authorization; (2) approval scope enforcement — no approval can authorize operations outside its declared scope; (3) time-delay bypass impossibility — no sequence of consensus-valid transactions can cause a pending transfer to release before its delay window; (4) guardian modification delay — no governance action can reduce the guardian modification delay for any account without that account's own explicit authorization.
Level 4 — External Security Audit
Before testnet launch, a third-party security audit must be commissioned for: the SynQ compiler and type checker (with specific focus on linear type enforcement and authority declaration soundness), the consensus-layer enforcement modules (approval validation, time-delay state machine, encrypted mempool decryption), and the Synergy Wallet's three-plane model and Local Policy Engine. The audit scope must explicitly include adversarial tests for each of the 13 attack categories in this document. Audit findings are classified as Critical, High, Medium, or Low. No Critical or High findings may remain unresolved at testnet launch.
SECURITY COMMITMENT:  The total addressable attack surface documented across all 13 categories exceeds $10 billion in confirmed losses. A protocol that eliminates even 70% of these vectors — not through education, not through tooling, but through architectural design — represents a generational improvement in blockchain security. The measures in this document are designed to eliminate the structural conditions that make these attacks possible, not merely to make them harder.
